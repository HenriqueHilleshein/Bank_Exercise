# Bank Exercise Application Development Agent

## Project Architecture

The Bank Exercise application follows a clean, layered architecture:

```
┌─────────────────────────────────────────┐
│     REST API Server (Server.h)          │
│  - HTTP endpoints                       │
│  - Request routing                      │
│  - Response serialization               │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│   Account Service (AccountService.h)    │
│  - Business logic                       │
│  - Account operations                   │
│  - Transaction management               │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│  Database Interface (IDBConnection.h)   │
│  - Abstract interface                   │
│  - Database contract                    │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│ FakeDBConnection (FakeDBConnection.h)   │
│  - In-memory database                   │
│  - Transaction support                  │
│  - Account storage                      │
└─────────────────────────────────────────┘
```

## Core Components

### 1. Account Hierarchy

**Base Class: Account**
- Manages balance
- Provides deposit/withdraw operations
- Stores account ID

**Derived Classes:**
- **CustomerAccount** - Personal accounts
  - Stores CustomerInformation (firstName, lastName)
  - Returns JSON with customerInformation field
  
- **EnterpriseAccount** - Business accounts
  - Stores EnterpriseInformation (YTunnus, companyName)
  - Returns JSON with enterpriseInformation field

**Pattern:**
```cpp
class Account {
    double balance;
    AccountId id;
    
    virtual nlohmann::json getAccountDetails() = 0;
};

class CustomerAccount : public Account {
    CustomerInformation info;
    nlohmann::json getAccountDetails() override;
};
```

### 2. Service Layer: AccountService

Implements business logic and orchestrates operations:

- **createCustomerAccount()** - Create customer accounts
- **createEnterpriseAccount()** - Create enterprise accounts
- **getAccount()** - Retrieve account by ID
- **depositToAccount()** - Add funds
- **withdrawFromAccount()** - Remove funds
- **transferBetweenAccounts()** - Move funds between accounts
- **getAccountIdWithCustomerInformation()** - Lookup by customer details
- **getAccountIdWithEnterpriseInformation()** - Lookup by enterprise details

**Key principles:**
- All operations throw `std::runtime_error` on failure
- Uses transactions for data consistency
- Database-agnostic (works with any IDBConnection implementation)

### 3. Database Interface: IDBConnection

Abstract interface defining database contract:

```cpp
class IDBConnection {
    virtual DBOperationResult getAccount(...) = 0;
    virtual DBOperationResult createCustomerAccount(...) = 0;
    virtual DBOperationResult createEnterpriseAccount(...) = 0;
    virtual std::unique_ptr<ITransaction> beginTransaction() = 0;
};
```

**Allows:**
- Multiple database implementations (Fake, SQL, MongoDB, etc.)
- Testability through mock/fake implementations
- Clean separation from business logic

### 4. Transaction Interface: ITransaction

Manages ACID properties:

```cpp
class ITransaction {
    virtual DBOperationResult getForUpdate(...) = 0;
    virtual DBOperationResult updateAccountBalance(...) = 0;
    virtual DBOperationResult commit() = 0;
    virtual DBOperationResult rollback() = 0;
};
```

**Features:**
- Row locking (getForUpdate)
- Isolated changes (rollback capability)
- Atomicity (commit/rollback)

### 5. REST API Server: Server

Exposes HTTP endpoints using cpp-httplib:

**Endpoints:**
- `GET /health` - Server health check
- `GET /account/{id}` - Get account details
- `POST /customer-account` - Create customer account
- `POST /enterprise-account` - Create enterprise account
- `POST /deposit` - Deposit to account
- `POST /withdraw` - Withdraw from account
- `POST /transfer` - Transfer between accounts
- `POST /customer-account-id` - Lookup customer by info
- `POST /enterprise-account-id` - Lookup enterprise by info

**Pattern:**
```cpp
svr.Get(R"(/account/(\d+))", [this](const httplib::Request& req, httplib::Response& res) {
    try {
        int id = std::stoi(req.matches[1]);
        auto account = _accountService->getAccount(id);
        res.set_content(account->getAccountDetails().dump(), "application/json");
    } catch (const std::exception& ex) {
        res.status = 404;
        res.set_content(nlohmann::json{{"error", ex.what()}}.dump(), "application/json");
    }
});
```

## Design Patterns Used

### 1. Strategy Pattern
Different account types implement the account interface:
- Strategy: Account type (Customer vs Enterprise)
- Context: AccountService uses any IAccount

### 2. Template Method Pattern
Account base class defines operation flow, derived classes provide details

### 3. Dependency Injection
`AccountService` receives `IDBConnection*` in constructor:
```cpp
AccountService::AccountService(IDBConnection* dbConnection)
    : _dbConnection(dbConnection) {}
```

### 4. Repository Pattern
`IDBConnection` acts as repository for account data

### 5. Transaction Pattern
`ITransaction` provides ACID guarantees

## Development Guidelines

### When Adding a New Feature

#### 1. Create the Interface (if needed)
Define abstract interface in `Include/`:
```cpp
class INewFeature {
public:
    virtual ~INewFeature() = default;
    virtual Result doSomething() = 0;
};
```

#### 2. Implement the Feature
Add implementation in `Source/`:
```cpp
class NewFeature : public INewFeature {
    Result doSomething() override { ... }
};
```

#### 3. Integrate with Service
Update `AccountService` or create new service

#### 4. Add API Endpoint
Register route in `Server::setupRoutes()`

#### 5. Write Tests
Create comprehensive unit tests before implementation (TDD)

### Naming Conventions

**Classes:**
- Interfaces: `I<Name>` (e.g., `IAccount`, `IDBConnection`)
- Implementations: `<Name>` or `<Name><Type>` (e.g., `Account`, `CustomerAccount`)
- Services: `<Name>Service` (e.g., `AccountService`)

**Methods:**
- Getters: `get<Property>()` (e.g., `getBalance()`)
- Setters: `set<Property>()` (e.g., `setBalance()`)
- Queries: `<Noun><Property>()` (e.g., `getAccountIdWithCustomerInformation()`)
- Actions: `<Verb><Noun>()` (e.g., `depositToAccount()`)

**Variables:**
- Member variables: `_camelCase` (e.g., `_accountService`)
- Local variables: `camelCase` (e.g., `accountId`)

**Files:**
- Headers: `Include/<ClassName>.h`
- Implementation: `Source/<ClassName>.cpp`
- Match class name to filename

### Error Handling

**Service Layer (AccountService):**
- Throw `std::runtime_error` with descriptive message
- Let caller decide how to handle

**Database Layer (IDBConnection):**
- Return `DBOperationResult` enum
- Specific error codes for different failures

**API Layer (Server):**
- Catch exceptions
- Set appropriate HTTP status codes (400, 404, 500)
- Return JSON error response

**Pattern:**
```cpp
// Service layer
if (!account) {
    throw std::runtime_error("Account not found");
}

// API layer
try {
    auto account = service->getAccount(id);
    res.set_content(...);
} catch (const std::exception& ex) {
    res.status = 404;
    res.set_content(nlohmann::json{{"error", ex.what()}}.dump(), ...);
}
```

### JSON Serialization

Use nlohmann::json for all JSON operations:

```cpp
// Creating JSON
nlohmann::json response{
    {"balance", account->getBalance()},
    {"accountId", accountId_to_json(account->getAccountId())}
};

// Parsing JSON
auto json = nlohmann::json::parse(req.body);
if (!json.contains("firstName")) {
    throw std::runtime_error("Missing firstName");
}
std::string firstName = json["firstName"].get<std::string>();
```

## Adding a New Account Type

### Step 1: Create Information Struct
```cpp
// Include/YourAccountInformation.h
struct YourAccountInformation {
    std::string field1;
    std::string field2;
};
```

### Step 2: Create Account Class
```cpp
// Include/YourAccount.h
class YourAccount : public Account {
public:
    YourAccount(double balance, const AccountId& id, const YourAccountInformation& info);
    nlohmann::json getAccountDetails() override;
private:
    YourAccountInformation _info;
};
```

### Step 3: Implement Class
```cpp
// Source/YourAccount.cpp
YourAccount::YourAccount(double balance, const AccountId& id, const YourAccountInformation& info)
    : Account(balance, id), _info(info) {}

nlohmann::json YourAccount::getAccountDetails() {
    return nlohmann::json{
        {"balance", getBalance()},
        {"accountId", accountId_to_json(getAccountId())},
        {"yourAccountInformation", {
            {"field1", _info.field1},
            {"field2", _info.field2}
        }}
    };
}
```

### Step 4: Add to Service
```cpp
// AccountService.h
AccountId createYourAccount(const YourAccountInformation info) const;

// AccountService.cpp
AccountId AccountService::createYourAccount(const YourAccountInformation info) const {
    AccountId accountId;
    DBOperationResult result = _dbConnection->createYourAccount(info, accountId);
    if (result != DBOperationResult::Success) {
        throw std::runtime_error("Failed to create your account");
    }
    return accountId;
}
```

### Step 5: Add to Database
```cpp
// FakeDBConnection.h
DBOperationResult createYourAccount(const YourAccountInformation info, AccountId& outAccountId) override;

// FakeDBConnection.cpp
DBOperationResult FakeDBConnection::createYourAccount(const YourAccountInformation info, AccountId& outAccountId) {
    // Implementation
}
```

### Step 6: Add API Endpoint
```cpp
// Server.cpp
svr.Post("/your-account", [this](const httplib::Request& req, httplib::Response& res) {
    try {
        auto json = nlohmann::json::parse(req.body);
        YourAccountInformation info{...};
        AccountId accountId = _accountService->createYourAccount(info);
        res.set_content(nlohmann::json{{"accountId", accountId_to_json(accountId)}}.dump(), ...);

```

## Extending the Database

To add a new database implementation (e.g., SQL):

### Step 1: Implement IDBConnection
```cpp
class SQLDBConnection : public IDBConnection {
    DBOperationResult getAccount(...) override;
    DBOperationResult createCustomerAccount(...) override;
    std::unique_ptr<ITransaction> beginTransaction() override;
};
```

### Step 2: Implement ITransaction
```cpp
class SQLTransaction : public ITransaction {
    DBOperationResult getForUpdate(...) override;
    DBOperationResult commit() override;
    DBOperationResult rollback() override;
};
```

### Step 3: Swap Implementation
```cpp
// main.cpp
// FakeDBConnection db;  // Old
SQLDBConnection db;       // New

AccountService service(&db);
```

No other code changes needed due to dependency injection!

## Transaction Management Best Practices

### When to Use Transactions
- Any operation that modifies multiple entities
- Operations that must be atomic
- Transfers between accounts
- Complex updates

### Example: Safe Transfer
```cpp
void AccountService::transferBetweenAccounts(int from, int to, double amount) const {
    std::unique_ptr<ITransaction> transaction = _dbConnection->beginTransaction();
    
    std::unique_ptr<IAccount> fromAccount;
    transaction->getForUpdate(from, fromAccount);
    
    std::unique_ptr<IAccount> toAccount;
    transaction->getForUpdate(to, toAccount);
    
    if (!fromAccount->withdraw(amount)) {
        throw std::runtime_error("Insufficient funds");
    }
    toAccount->deposit(amount);
    
    transaction->updateAccountBalance(*fromAccount);
    transaction->updateAccountBalance(*toAccount);
    
    transaction->commit();  // All-or-nothing
}
```

## Performance Considerations

1. **Avoid unnecessary account lookups** - Cache when possible
2. **Lock accounts in consistent order** - Prevent deadlocks
3. **Keep transactions short** - Minimize lock duration
4. **Batch operations** - When applicable
5. **Index frequently-queried fields** - YTunnus, customer names
