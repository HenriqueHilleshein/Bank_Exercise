# Bank Exercise Unit Testing Agent

## Project Context

This is a C++ banking application with a layered architecture:

```
REST API Server (Server.h)
        ↓
Account Service (AccountService.h) - Business Logic
        ↓
Database Interface (IDBConnection.h)
        ↓
Fake DB Connection (FakeDBConnection.h) - In-Memory Implementation
```

### Key Components to Test

1. **Account Classes** (`Account.h`, `CustomerAccount.h`, `EnterpriseAccount.h`)
   - Basic balance operations (deposit, withdraw)
   - Account ID management
   - JSON serialization for different account types

2. **Database Layer** (`FakeDBConnection.h`)
   - CRUD operations for accounts
   - Transaction management (begin, commit, rollback)
   - Account lookups by ID and customer/enterprise information

3. **Business Logic** (`AccountService.h`)
   - Account creation
   - Deposit and withdrawal operations
   - Money transfers between accounts
   - Error handling and validation

## Testing Patterns Used

### 1. Test Fixture Pattern
All test suites use Google Test fixtures for setup and teardown:

```cpp
class AccountTests : public ::testing::Test {
protected:
    AccountId testAccountId{1000, 1609459200000000000};
    double initialBalance = 100.0;
    
    // Setup runs before each test
    // Teardown runs after each test
};
```

### 2. Assertion Pattern
Use appropriate assertions:
- `EXPECT_EQ(actual, expected)` - Equality checks
- `EXPECT_DOUBLE_EQ(actual, expected)` - Floating point equality
- `EXPECT_TRUE(condition)` / `EXPECT_FALSE(condition)` - Boolean checks
- `EXPECT_NE(actual, expected)` - Not equal
- `EXPECT_THROW(statement, exception_type)` - Exception checking
- `EXPECT_EQ(ptr, nullptr)` - Null pointer checks

### 3. Test Organization
Each test suite focuses on a single class:
- **AccountTests** - Base Account class behavior
- **CustomerAccountTests** - Customer-specific functionality
- **EnterpriseAccountTests** - Enterprise-specific functionality
- **FakeDBConnectionTests** - Database operations and transactions
- **AccountServiceTests** - Business logic and workflows

### 4. Test Coverage Strategy

#### For Account Operations:
- **Happy path**: Normal deposit, withdraw, balance queries
- **Edge cases**: Zero amounts, decimal precision, large numbers
- **Error cases**: Insufficient funds, negative balances
- **State verification**: Check balance after each operation

#### For Database Operations:
- **CRUD**: Create, read, update operations
- **Transactions**: Commit and rollback behavior
- **Isolation**: Changes within transactions don't affect other connections
- **Validation**: Invalid input handling

#### For Business Logic:
- **Workflows**: Multi-step operations (create → deposit → withdraw → transfer)
- **Error propagation**: Exceptions bubble up correctly
- **Cross-account operations**: Transfers between different account types
- **Edge cases**: Same account transfers, zero transfers, max amounts

## Writing New Unit Tests

### Step 1: Create Test File
New test files follow the pattern: `*Tests.cpp`

Example: `NewFeatureTests.cpp`

### Step 2: Include Necessary Headers
```cpp
#include <gtest/gtest.h>
#include <YourClass.h>
#include <YourInterface.h>
```

### Step 3: Create Test Fixture
```cpp
class YourClassTests : public ::testing::Test {
protected:
    // Member variables for setup
    YourClass instance;
    
    // Optional: Setup method
    void SetUp() override {
        // Called before each test
    }
    
    // Optional: Teardown method
    void TearDown() override {
        // Called after each test
    }
};
```

### Step 4: Write Test Cases
```cpp
TEST_F(YourClassTests, DescriptiveTestName) {
    // Arrange: Set up test data
    int input = 5;
    
    // Act: Perform the operation
    int result = instance.someMethod(input);
    
    // Assert: Verify the result
    EXPECT_EQ(result, 10);
}
```

### Step 5: Update CMakeLists.txt
Add your test file to the `Bank_Exercise_Tests` executable:

```cmake
add_executable(Bank_Exercise_Tests
    Tests/NewFeatureTests.cpp
    # ... other test files
)
```

## Test Naming Convention

Test names follow this pattern: `[Feature][Scenario][ExpectedResult]`

Examples:
- `DepositIncreasesBalance` - Feature: Deposit, Scenario: Normal operation, Result: Balance increases
- `WithdrawInsufficientFundsReturnsFalse` - Feature: Withdraw, Scenario: Insufficient funds, Result: Returns false
- `TransferThrowsOnNonExistentAccount` - Feature: Transfer, Scenario: Account doesn't exist, Result: Throws exception

## Coverage Requirements

### Aim for 100% code coverage including:

1. **Normal execution paths** (happy path)
   ```cpp
   TEST_F(AccountTests, DepositIncreasesBalance) {
       account.deposit(50.0);
       EXPECT_DOUBLE_EQ(account.getBalance(), 150.0);
   }
   ```

2. **Error paths** (exceptions and error returns)
   ```cpp
   TEST_F(AccountTests, WithdrawInsufficientFunds) {
       bool success = account.withdraw(200.0);
       EXPECT_FALSE(success);
   }
   ```

3. **Edge cases** (boundary conditions)
   ```cpp
   TEST_F(AccountTests, WithdrawZeroAmount) {
       account.withdraw(0.0);
       EXPECT_DOUBLE_EQ(account.getBalance(), 100.0);
   }
   ```

4. **State transitions** (multiple operations)
   ```cpp
   TEST_F(AccountTests, DepositThenWithdraw) {
       account.deposit(50.0);
       account.withdraw(30.0);
       EXPECT_DOUBLE_EQ(account.getBalance(), 120.0);
   }
   ```

## Testing Database and Transactions

### Transaction Testing Pattern:
```cpp
TEST_F(FakeDBConnectionTests, TransactionRollbackRestoresState) {
    // Create account with initial state
    CustomerInformation info{"John", "Doe"};
    AccountId accountId;
    db.createCustomerAccount(info, accountId);
    
    // Get initial state
    std::unique_ptr<IAccount> accountBefore;
    db.getAccount(accountId.UniqueIdentifier, accountBefore);
    double balanceBefore = accountBefore->getBalance();
    
    // Begin transaction and modify
    std::unique_ptr<ITransaction> transaction = db.beginTransaction();
    std::unique_ptr<IAccount> account;
    transaction->getForUpdate(accountId.UniqueIdentifier, account);
    account->deposit(100.0);
    transaction->updateAccountBalance(*account);
    
    // Rollback and verify restoration
    transaction->rollback();
    
    std::unique_ptr<IAccount> accountAfter;
    db.getAccount(accountId.UniqueIdentifier, accountAfter);
    EXPECT_DOUBLE_EQ(accountAfter->getBalance(), balanceBefore);
}
```

## Testing Business Logic Workflows

Test complex scenarios involving multiple operations:

```cpp
TEST_F(AccountServiceTests, TransferBetweenAccountsWorks) {
    // Setup: Create two accounts
    CustomerInformation info1{"John", "Doe"};
    CustomerInformation info2{"Jane", "Smith"};
    AccountId id1 = service.createCustomerAccount(info1);
    AccountId id2 = service.createCustomerAccount(info2);
    
    // Act: Deposit and transfer
    service.depositToAccount(id1.UniqueIdentifier, 1000.0);
    service.transferBetweenAccounts(id1.UniqueIdentifier, id2.UniqueIdentifier, 300.0);
    
    // Assert: Verify both accounts
    std::unique_ptr<IAccount> account1 = service.getAccount(id1.UniqueIdentifier);
    std::unique_ptr<IAccount> account2 = service.getAccount(id2.UniqueIdentifier);
    EXPECT_DOUBLE_EQ(account1->getBalance(), 700.0);
    EXPECT_DOUBLE_EQ(account2->getBalance(), 300.0);
}
```

## Error Handling Tests

Always test error conditions:

```cpp
// Test exception throwing
TEST_F(AccountServiceTests, DepositThrowsOnNonExistentAccount) {
    EXPECT_THROW(service.depositToAccount(9999, 100.0), std::runtime_error);
}

// Test error return values
TEST_F(FakeDBConnectionTests, CreateAccountWithEmptyNameReturnsError) {

    AccountId accountId;
    
    DBOperationResult result = db.createCustomerAccount(info, accountId);
    
    EXPECT_EQ(result, DBOperationResult::InvalidInput);
}
```

## Best Practices

1. **One assertion focus per test**: Each test should verify one behavior
2. **Clear test names**: Test name should explain what is being tested
3. **Arrange-Act-Assert**: Structure tests with setup, execution, verification
4. **No test interdependencies**: Tests should be independent and run in any order
5. **Test data isolation**: Each test uses its own data via fixtures
6. **Meaningful assertions**: Use specific assertions, not just boolean checks
7. **Document complex tests**: Add comments explaining tricky test setups
8. **Mock external dependencies**: Use fake implementations (like FakeDBConnection)

## Common Issues and Solutions

### Tests failing after code changes
- Verify the implementation matches the test expectations
- Check for state isolation issues between tests
- Ensure transaction cleanup (commit/rollback)

### Memory leaks in tests
- Use `std::unique_ptr` for automatic cleanup
- Verify test fixtures clean up properly
- Check for circular references

### Flaky tests
- Avoid time-dependent operations
- Don't rely on test execution order
- Ensure proper transaction isolation
